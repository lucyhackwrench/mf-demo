# Эксперимент 06: Оптимизация загрузки UI-библиотеки

## Цель

Разобраться, можно ли не загружать весь бандл `@demo/ui` целиком, а получать только реально используемые части — и как это соотносится с MF shared scope.

Конкретные вопросы:

1. Что сейчас попадает в бандл `@demo/ui` — весь модуль или только используемые экспорты?
2. Работает ли tree shaking для shared-библиотеки в MF — убирает ли rspack неиспользуемые экспорты из shared scope?
3. Что такое granular shared: регистрация отдельных путей вместо всего пакета (`'@demo/ui/Button'` вместо `'@demo/ui'`) — решает ли это проблему?
4. Влияет ли `sideEffects: false` в `package.json` ui-библиотеки на объём бандла?
5. Как соотносятся подходы: shared целиком vs granular shared vs code splitting внутри библиотеки?


Пока предполагаю, что ключевое ограничение: shared scope отдаёт модуль целиком. MF не умеет отдавать «только Button» — он регистрирует фабрику для всего @demo/ui и кешируент весь объект модуля. Но теоретически можно разбить с помомщью granular shared.
